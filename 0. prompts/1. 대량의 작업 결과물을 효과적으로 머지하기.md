  ## 1) 작업 핵심 목표 및 범위

  - 목표: `docs/0.PRD_v1_opus46_fn.md`, `tasks/0.TASKS_LIST_v0_2_opus47_fn.md`, 현재 GitHub Issue 및 PR 진행상태를 근거
  로, 후속 에이전틱 루프에서 한 번에 안전하게 처리 가능한 최대 작업 묶음을 선정하고 구현·검증·보고까지 완료한다.
  - 시작 지점: 현재 브랜치와 현재 워크트리 상태에서 시작한다. 먼저 `git status --short`, `git branch --show-current`,
  `gh issue list`, `gh pr list` 결과를 확인하고 대화에 요약한다.
  - 작업 대상:
    - 문서 기준: `docs/0.PRD_v1_opus46_fn.md`, `tasks/0.TASKS_LIST_v0_2_opus47_fn.md`
    - 진행상태 기준: 현재 저장소의 GitHub Issue, PR, 연결된 브랜치 및 체크 상태
    - 구현 범위: 문서와 Issue/PR 상태상 Phase-0 후속 작업으로 묶어서 처리 가능한 항목 중, 의존성이 명확하고 검증 가능한
    최대 범위
    - 기록 범위: 작업 중 발견한 미확정 의사결정사항, 구현 근거, 검증 결과, 남은 작업
  - 작업 자율성:
    - 기존 문서와 Issue/PR에서 명확히 정의된 작업은 사용자 추가 승인 없이 진행한다.
    - 기존 문서에서 명확하게 정하지 않은 핵심 의사결정사항이 필요하면 임의로 확정하지 말고 결정 체크포인트에 기록한다.
    - `main` 직접 푸시, 기존 PR 강제 변경, 배포 트리거, 외부 유료 리소스 사용, 스키마/인증/권한 정책의 의미 변경은 사용
    자 확인 없이 진행하지 않는다.

  ## 2) 작업 세부 규칙

  - 시작 시 반드시 다음 자료를 읽고 기준점을 만든다:
    - `AGENTS.md`
    - `docs/0.PRD_v1_opus46_fn.md`
    - `tasks/0.TASKS_LIST_v0_2_opus47_fn.md`
    - 현재 GitHub Issue 목록
    - 현재 GitHub PR 목록
    - 현재 브랜치와 워크트리 상태
  - GitHub 상태 확인은 가능한 경우 `gh` CLI를 사용한다:
    - `gh issue list --limit 100 --state all`
    - `gh pr list --limit 100 --state all`
    - 필요한 경우 관련 Issue/PR 본문과 코멘트를 추가 확인한다.
  - 먼저 “이번 루프에서 한 번에 처리할 최대 작업 범위”를 산정한다:
    - 문서상 Phase-0 범위에 속할 것
    - 선행 의존성이 충족되어 있을 것
    - 검증 명령 또는 실제 사용 시나리오로 완료 여부를 증명할 수 있을 것
    - 하나의 루프에서 구현·테스트·수동 QA·보고까지 끝낼 수 있을 것
  - 작업은 작은 단위로 진행하되, 같은 사용자 가치 또는 같은 Issue/PR 완료 조건에 속한 항목은 한 묶음으로 처리한다.
  - 모든 구현 전에는 기존 동작과 요구사항을 확인하고, 가능한 경우 실패하는 테스트나 재현 절차를 먼저 확보한다.
  - 모든 변경 후에는 최소한 다음을 수행한다:
    - 변경 파일에 대한 타입체크/LSP 진단 또는 프로젝트 표준 검증
    - 관련 테스트 실행
    - 사용자 관점의 실제 표면 검증: 브라우저, HTTP call, CLI, DB 상태 확인 중 해당 작업에 맞는 방식
  - 의사결정 체크포인트 문서를 반드시 생성 또는 갱신한다:
    - 경로: `docs/goals/agentic-loop-decision-checkpoint.md`
    - 문서에는 아래 카운터를 grep 가능한 형태로 유지한다:
      - `CORE_DECISION_COUNT: <number>`
      - `MINOR_DECISION_COUNT: <number>`
    - 각 의사결정사항은 다음 필드를 포함한다:
      - ID: `CORE-001` 또는 `MINOR-001`
      - 분류: `CORE` 또는 `MINOR`
      - 발견 위치: 문서/Issue/PR/파일 경로
      - 미확정 내용
      - 왜 이번 루프에서 임의 확정하면 위험한지
      - 임시 처리 방식: 보류 / 보수적 기본값 / 범위 제외
  - 핵심 의사결정사항(`CORE`)은 다음 중 하나에 해당하는 미확정 사항이다:
    - 제품 흐름, 인증/권한, 데이터 모델, API 계약, 비용 게이트, Phase 경계, 보안 정책, 관리자 승인 정책, 이메일 실패 처
    리, 사용자 공개 범위를 바꾸는 결정
  - 부수 의사결정사항(`MINOR`)은 다음 중 하나에 해당하는 미확정 사항이다:
  - 기존 문서의 명확한 결정을 뒤집지 않는다. 특히 AGENTS.md의 Product Decisions T1~T10을 우선 준수한다.
  - 코드 변경이 발생하면 작업 단위별로 검증 결과를 남긴다. 커밋은 사용자가 명시적으로 요청한 경우에만 생성하고, 그렇지
  않으면 커밋 메시지 초안만 제시한다.

  ## 3) 종료 조건 및 종료 방법

  - 종료 조건 (아래 중 하나라도 충족되는 순간 루프를 즉시 멈춘다):
    - 선정한 이번 루프 작업 범위의 필수 구현·검증·보고가 완료됨 → STOP REASON: WORK_SCOPE_COMPLETE
    - 기존 문서에서 명확하게 정해놓지 않은 핵심 의사결정사항이 3가지 이상 누적됨 (`CORE_DECISION_COUNT >= 3`) → STOP
    REASON: CORE_DECISION_LIMIT
    - 기존 문서에서 명확하게 정해놓지 않은 부수적인 의사결정사항이 10가지 이상 누적됨 (`MINOR_DECISION_COUNT >= 10`) →
    STOP REASON: MINOR_DECISION_LIMIT
    - 같은 검증 실패 또는 같은 구현 blocker가 3회 반복됨 → STOP REASON: REPEATED_FAILURE
    - 평가-진행 라운드(turn) 누적 40회 도달 → STOP REASON: TURN_CAP (= or stop after 40 turns). 여기서 turn은 평가자가
    한 번 점검하는 메인 에이전트 응답 사이클을 뜻한다.
  - 종료 방법:
    1. `docs/goals/agentic-loop-decision-checkpoint.md`의 `CORE_DECISION_COUNT`와 `MINOR_DECISION_COUNT`를 최종 갱신하
    고, 종료 사유를 `STOP_REASON: <CODE>` 형식으로 기록한다.
    2. 다음 명령 또는 동등한 확인 명령의 출력을 대화에 surface한다:
       - `git status --short`
       - `grep -n "CORE_DECISION_COUNT\\|MINOR_DECISION_COUNT\\|STOP_REASON" docs/goals/agentic-loop-decision-
       checkpoint.md`
       - 관련 테스트/타입체크/린트 명령의 exit code와 핵심 출력
    3. GitHub Issue/PR 기준으로 처리된 항목, 보류된 항목, 다음 루프 후보를 요약한다.
    4. 구현 변경이 있었다면 변경 파일, 검증 명령, 실제 표면 QA 증거를 함께 보고한다.
    - 사용자 승인 없는 기존 PR 머지 또는 종료
    - 사용자 승인 없는 배포 트리거
    - 사용자 승인 없는 유료 외부 API 호출 또는 비용 발생 작업
    - 실패 테스트 삭제, skip 처리, 기대값 약화로 인한 통과 처리
    - 타입 오류를 숨기기 위한 `as any`, `@ts-ignore`, `@ts-expect-error` 추가
  - 수정 금지 또는 주의 파일:
    - `docs/0.PRD_v1_opus46_fn.md`와 `tasks/0.TASKS_LIST_v0_2_opus47_fn.md`는 원칙적으로 기준 문서로만 사용한다.
    - 단, 명백한 오탈자 또는 이번 작업 결과를 반영해야 하는 추적 문서 갱신이 필요한 경우에는 변경 전 이유를 기록한다.
    - 의사결정 체크포인트 문서 `docs/goals/agentic-loop-decision-checkpoint.md`는 예외적으로 생성/수정 가능하다.
  - 활성 범위 외 변경 금지:
    - 이번 루프에서 선정한 Issue/PR/문서 요구사항과 직접 관련 없는 리팩터링, 디자인 변경, 의존성 업그레이드, 설정 변경은
    하지 않는다.
    - 발견한 추가 개선점은 구현하지 말고 다음 루프 후보로 기록한다.
  - 완료 보고는 반드시 한국어로 작성한다.

      - 임시 처리 방식: 보류 / 보수적 기본값 / 범위 제외
  - 핵심 의사결정사항(`CORE`)은 다음 중 하나에 해당하는 미확정 사항이다:
    - 제품 흐름, 인증/권한, 데이터 모델, API 계약, 비용 게이트, Phase 경계, 보안 정책, 관리자 승인 정책, 이메일 실패 처
    리, 사용자 공개 범위를 바꾸는 결정
  - 부수 의사결정사항(`MINOR`)은 다음 중 하나에 해당하는 미확정 사항이다:
  - 기존 문서의 명확한 결정을 뒤집지 않는다. 특히 AGENTS.md의 Product Decisions T1~T10을 우선 준수한다.
  - 코드 변경이 발생하면 작업 단위별로 검증 결과를 남긴다. 커밋은 사용자가 명시적으로 요청한 경우에만 생성하고, 그렇지
  않으면 커밋 메시지 초안만 제시한다.

  ## 3) 종료 조건 및 종료 방법

  - 종료 조건 (아래 중 하나라도 충족되는 순간 루프를 즉시 멈춘다):
    - 선정한 이번 루프 작업 범위의 필수 구현·검증·보고가 완료됨 → STOP REASON: WORK_SCOPE_COMPLETE
    - 기존 문서에서 명확하게 정해놓지 않은 핵심 의사결정사항이 3가지 이상 누적됨 (`CORE_DECISION_COUNT >= 3`) → STOP
    REASON: CORE_DECISION_LIMIT
    - 기존 문서에서 명확하게 정해놓지 않은 부수적인 의사결정사항이 10가지 이상 누적됨 (`MINOR_DECISION_COUNT >= 10`) →
    STOP REASON: MINOR_DECISION_LIMIT
    - 같은 검증 실패 또는 같은 구현 blocker가 3회 반복됨 → STOP REASON: REPEATED_FAILURE
    - 평가-진행 라운드(turn) 누적 40회 도달 → STOP REASON: TURN_CAP (= or stop after 40 turns). 여기서 turn은 평가자가
    한 번 점검하는 메인 에이전트 응답 사이클을 뜻한다.
  - 종료 방법:
    1. `docs/goals/agentic-loop-decision-checkpoint.md`의 `CORE_DECISION_COUNT`와 `MINOR_DECISION_COUNT`를 최종 갱신하
    고, 종료 사유를 `STOP_REASON: <CODE>` 형식으로 기록한다.
    2. 다음 명령 또는 동등한 확인 명령의 출력을 대화에 surface한다:
       - `git status --short`
       - `grep -n "CORE_DECISION_COUNT\\|MINOR_DECISION_COUNT\\|STOP_REASON" docs/goals/agentic-loop-decision-
       checkpoint.md`
       - 관련 테스트/타입체크/린트 명령의 exit code와 핵심 출력
    3. GitHub Issue/PR 기준으로 처리된 항목, 보류된 항목, 다음 루프 후보를 요약한다.
    4. 구현 변경이 있었다면 변경 파일, 검증 명령, 실제 표면 QA 증거를 함께 보고한다.
    - 사용자 승인 없는 기존 PR 머지 또는 종료
    - 사용자 승인 없는 배포 트리거
    - 사용자 승인 없는 유료 외부 API 호출 또는 비용 발생 작업
    - 실패 테스트 삭제, skip 처리, 기대값 약화로 인한 통과 처리
    - 타입 오류를 숨기기 위한 `as any`, `@ts-ignore`, `@ts-expect-error` 추가
  - 수정 금지 또는 주의 파일:
    - `docs/0.PRD_v1_opus46_fn.md`와 `tasks/0.TASKS_LIST_v0_2_opus47_fn.md`는 원칙적으로 기준 문서로만 사용한다.
    - 단, 명백한 오탈자 또는 이번 작업 결과를 반영해야 하는 추적 문서 갱신이 필요한 경우에는 변경 전 이유를 기록한다.
    - 의사결정 체크포인트 문서 `docs/goals/agentic-loop-decision-checkpoint.md`는 예외적으로 생성/수정 가능하다.
  - 활성 범위 외 변경 금지:
    - 이번 루프에서 선정한 Issue/PR/문서 요구사항과 직접 관련 없는 리팩터링, 디자인 변경, 의존성 업그레이드, 설정 변경은
    하지 않는다.
    - 발견한 추가 개선점은 구현하지 말고 다음 루프 후보로 기록한다.
  - 완료 보고는 반드시 한국어로 작성한다.
